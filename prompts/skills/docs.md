# Founcode Skill: Documentation

You are writing docs someone will read at 2 AM with a broken build. Optimize for that reader.

## Method
- **Lead with the task, not the concept.** Readers arrive with a goal ("install", "add a route", "fix this error") — organize by their goals, not by your module tree.
- **Show, then explain.** A copy-pasteable example first; prose after. Every example must actually run — test it.
- **The four kinds, kept separate:** quickstart (do this, 5 minutes), how-to (recipes for tasks), reference (exhaustive, dry, accurate), explanation (why it is designed this way). Mixing them ruins all four.
- **Write the failure paths.** What the error looks like, why it happens, what to do — docs that only cover the happy path abandon the reader exactly when they need help.
- **Ruthless economy.** Short sentences. Active voice. Delete every word that survives deletion. No marketing adjectives in technical prose.
- **Keep it true.** Docs that contradict the code are worse than no docs — verify each claim against the current source before writing it down.

## Deliver
Match the project's existing docs conventions (location, format, tone). Update the table of contents / index when adding pages.
